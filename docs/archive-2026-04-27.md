# Part 1 ‚Äî Google Gemini TTS adapter

## Why

The user opened [`free-tts-api-roundup-2026-04-26.md`](free-tts-api-roundup-2026-04-26.md)
and decided Gemini was the next channel to wire up. AI Studio shows
two free TTS models on their account:

- `gemini-2.5-flash-preview-tts` ‚Äî 3 RPM / 10K TPM / 10 RPD
- `gemini-3.1-flash-tts-preview` ‚Äî 3 RPM / 10K TPM / 10 RPD
- `gemini-2.5-pro-preview-tts` ‚Äî 0/0 (not on free tier; excluded)

Per user direction, the rate limit isn't the adapter's problem ‚Äî if
3 RPM bites users, they figure it out. So no client-side throttling
or queuing.

## Adapter shape

`gemini_tts_adapter.dart` is a single REST adapter:

- **Endpoint**: `POST {baseUrl}/v1beta/models/{model}:generateContent`
- **Auth**: `x-goog-api-key: {apiKey}` header
- **Modes**: presetVoice (30 prebuilt voices, hard-coded), with an
  optional natural-language style instruction prepended to the text
  (Gemini has no separate style field). Clone-from-ref-audio is
  unsupported and the adapter throws.
- **Speed**: not natively controllable, so `request.speed != 1.0` is
  encoded as `"speak quickly"` / `"speak slowly"` in the prepended
  instruction.
- **Response**: base64 PCM in `candidates[0].content.parts[0].inlineData.data`,
  mime `audio/L16;codec=pcm;rate=24000`. Adapter parses the rate from
  the mime type, falls back to 24000, and wraps in a minimal RIFF/WAVE
  container before returning so `TtsResult.contentType` is `audio/wav`
  and the audio player needs no decoder change.

The 30 prebuilt voices and the 2 free model IDs are hard-coded in
`kPrebuiltVoices` / `kModels` because AI Studio doesn't expose a
filtered list endpoint. `getModels()` and `getSpeakers()` return
copies of those lists so the existing model+voice cache pipeline
works without a network call.

Error decoding parses Google's `{error: {code, status, message}}`
shape so the surfaced error reads e.g. `HTTP 404: 404 / NOT_FOUND /
models/X is not found ‚Ä¶` rather than a raw JSON dump.

**File**: `lib/data/adapters/gemini_tts_adapter.dart` (new).

# Part 2 ‚Äî Two follow-up fixes after first smoke test

## Wrong 3.1 model ID

The user picked Gemini 3.1 in the picker and got
`HTTP 404: NOT_FOUND / models/gemini-3.1-flash-preview-tts is not
found for API version v1beta`.

I'd guessed the 3.1 ID by mirroring the 2.5 naming convention
(`-flash-preview-tts`). Google actually reordered the suffix:

- 2.5: `gemini-2.5-flash-preview-tts` (suffix `-preview-tts`)
- 3.1: `gemini-3.1-flash-tts-preview` (suffix `-tts-preview`)

Confirmed against the official docs page
`ai.google.dev/gemini-api/docs/models/gemini-3.1-flash-tts-preview`.

Patched `kModels` in `gemini_tts_adapter.dart`. Voice list and
request shape are identical between 2.5 and 3.1, so the adapter
itself didn't change.

**File**: `lib/data/adapters/gemini_tts_adapter.dart`.

## `deleteVoiceAsset` silently failing on FK constraint

The user deleted a Gemini-bound character that had hit the 404, then
tried to recreate it with the same name on the 2.5 model. The create
dialog rejected with "A character with this name already exists" ‚Äî
even though the user had clicked Delete a moment earlier.

Root cause chain:

1. Three tables have non-nullable FK references to `VoiceAssets.id`:
   - `voice_bank_members.voiceAssetId`
   - `tts_jobs.voiceAssetId`
   - `quick_tts_histories.voiceAssetId`

   (The phase / dialog / video_dub segment tables also reference
   `voiceAssetId`, but those columns are `.nullable()` and not
   declared as FKs in the Drift schema, so they don't constrain
   the delete.)

2. Every newly-created character is auto-added to the default voice
   bank (`voiceBankMembers`), so every delete attempt has at least
   one row to violate.

3. `deleteVoiceAsset` was a single `delete().go()` with no
   transaction and no child cleanup. SQLite raised
   `FOREIGN KEY constraint failed`, the future rejected with that
   exception, but the button handler awaited it without try/catch ‚Äî
   so the exception bubbled into the async gap and Flutter swallowed
   it. The asset stayed in `voice_assets`, the row count returned 0,
   and the UI proceeded as if delete had succeeded.

4. The next "Create Character" call reads `voiceAssetsStreamProvider`
   fresh (the provider re-emits on the next Drift query, which still
   sees the row), and the `existingAssets.any((a) => a.name == name)`
   pre-check at line 1656 rejected the create.

Two-part fix:

- **`deleteVoiceAsset` now runs in a transaction** that wipes the
  three child tables before the parent delete, mirroring the
  existing `deleteBank` pattern at line 329.
- **The Delete button handler now wraps the call in try/catch** and
  surfaces the error as a snackbar. If a future schema change adds
  another non-nullable FK, the user sees the failure immediately
  instead of getting a phantom-delete state.

**Files**: `lib/data/database/app_database.dart`,
`lib/presentation/screens/voice_character_screen.dart`.

## Recovery for already-stuck users

The fix only protects future deletes. Anyone who hit this bug before
the fix has an orphan asset in `voice_assets`. They can either delete
it now (the new code handles it correctly) or rename it. No migration
needed.
