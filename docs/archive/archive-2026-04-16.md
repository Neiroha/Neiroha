## Summary

Bug-fix session focused on CosyVoice Native adapter synthesis failures (Clone + Instruct modes) plus

## Root Cause: CosyVoice 400 "未找到角色"

Both `zero_shot` (Clone) and `instruct` mode syntheses were failing with HTTP 400.

**Cause:** The Flutter adapter was sending `profile=<presetVoiceName>` in the multipart upload
request. The server's `build_runtime_char_config` does a strict lookup:

```python
base_config = character_config.get_character(requested_character)
if not base_config:
    raise ValueError(f"未找到角色: {requested_character}")
```

Users were typing arbitrary text into the "Profile Name (optional)" field in the character creation
dialog not realising it maps to a server-side profile registry, not a local label. The server
rejected every upload that carried a non-registered profile name.

## Changes

### `lib/data/adapters/cosyvoice_adapter.dart`

- **Removed `profile` from `_synthesizeWithUpload`** — when uploading audio the upload fully
  defines the voice; sending an unverified profile name causes 400 regardless of the audio content.
  Profile is still sent in `_synthesizeJson` (no-upload path) where the server needs it to locate
  stored reference audio.
- **Added `getProfiles()` method** — fetches `/cosyvoice/profiles`, returns `List<CosyVoiceProfile>`.
  The character creation dialog calls this to present a validated dropdown.
- **Added `CosyVoiceProfile` data class** — `id`, `name`, `mode`, `modeLabel`.

### `lib/presentation/screens/voice_character_screen.dart`

- **Added `cosyvoice_adapter.dart` import** for `CosyVoiceAdapter` / `CosyVoiceProfile` types.
- **Added state fields** `_cosyProfiles`, `_loadingCosyProfiles`, `_selectedCosyProfileId` to
  `_CreateCharacterDialogState`.
- **Added `_fetchCosyVoiceProfiles()`** — called from `_fetchModelsAndSpeakers()` when CosyVoice
  adapter is selected; casts the adapter to `CosyVoiceAdapter` and calls `getProfiles()`.
- **Replaced free-text "Profile Name" field** with a `DropdownButtonFormField` populated from the
  live profile list. Default item is `null` → "None (use uploaded audio only)". Each item shows
  `name · modeLabel` to disambiguate profiles with the same name but different modes.
- Added descriptive sub-label explaining that only server-registered profiles may be sent.

### `docs/api.md`

- Updated CosyVoice section: added `GET /cosyvoice/profiles` row, corrected multipart field list,
  added **Important** callout about why `profile` is omitted from multipart.
- Added `CosyVoiceProfile` data class docs.
- Added planned adapters table (Fish Speech, Kokoro, F5-TTS) with link to `add-llm-tts-adapter.md`.

### `docs/add-llm-tts-adapter.md` (new)

Comprehensive guide for an LLM to add a new TTS backend from scratch, covering:
- The three universal synthesis modes and how they map to `TaskMode` / `TtsRequest` fields
- All required touch points (adapter class, enum, factory, UI dialog)
- Concrete annotated examples
- Decision tree for common backend patterns

## Architecture Notes (unchanged)

- `VoiceAsset.modelName` stores the CosyVoice synthesis mode string (`zero_shot` / `cross_lingual`
  / `instruct`). `createAdapter(provider, modelName: asset.modelName)` passes it through.
- `TaskMode` enum: `presetVoice` | `cloneWithPrompt` | `voiceDesign` — maps to adapter behaviour,
  not to adapter type.
- The upload vs JSON path decision in `CosyVoiceAdapter.synthesize()` is driven solely by whether
  `request.refAudioPath` is non-null.
