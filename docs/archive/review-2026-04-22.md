## 🔴 Waveform extraction OOMs on long videos

**Location:** `lib/data/storage/ffmpeg_service.dart:101-123`
(`extractWaveformPeaks`).

**Problem:** `Process.run` buffers the entire stdout. At 8 kHz mono
PCM16:
- 2-hour video → ~115 MB in memory
- 4-hour video → ~230 MB

Any production-length input is a potential OOM.

**Fix:** switch to `Process.start` + a streaming peak reducer that
consumes the bytes as they arrive and updates a running per-bucket
max. Also enables mid-extract cancellation (nice-to-have for users
who change videos while a big extract is in flight).

Sketch:

```dart
final process = await Process.start(path, ['-v','error', /* … */]);
final peaks = List<double>.filled(bucketCount, 0.0);
var sampleIdx = 0;
final bucketSize = expectedSampleCount / bucketCount; // precomputed from ffprobe
int current = 0, max = 0;
await for (final chunk in process.stdout) {
  for (var i = 0; i + 1 < chunk.length; i += 2) {
    final v = _le16Signed(chunk[i], chunk[i + 1]).abs();
    if (v > max) max = v;
    sampleIdx++;
    if (sampleIdx / bucketSize > current + 1) {
      peaks[current] = max / 32768.0;
      current++;
      max = 0;
    }
  }
}
```

Caveat: streaming needs the sample count up front to size buckets.
Either `ffprobe -show_entries format=duration` first (adds one more
child process), or use a two-pass approach where pass 1 counts and
pass 2 reduces. The ffprobe approach plays nicely with PR candidate
(4) in `plan.md`.

## 🟡 `_onVideoTick` is not reentrancy-safe

**Location:** `lib/presentation/screens/video_dub_screen.dart:285-329`.

**Problem:** The media_kit position stream fires ~30×/sec. The tick
handler is async; its `stop → play → seek` sequence can overlap
between ticks. The `active?.id == _activeCueId` early-return catches
the common case but there's still a window where a seek from tick
N+1 lands on tick N's old play call — manifests as a cue replaying
from zero, or silence where a cue should be audible.

**Fix:** one-flight guard.

```dart
bool _ticking = false;
Future<void> _onVideoTick(Duration pos) async {
  if (_ticking) return;
  _ticking = true;
  try {
    /* existing body */
  } finally {
    _ticking = false;
  }
}
```

5 lines. Prevents an intermittent glitch that'd be painful to
reproduce if reported later.

## 🟡 `_pxToMs` uses stale `context.size`

**Location:** `lib/presentation/widgets/video_dub_timeline.dart:618-624`.

**Problem:** `context.size?.width` is only populated after the first
layout pass. First ruler tap after the editor opens computes body
width as `0 - _headerWidth`, returns `_viewLeftMs` with no x offset,
and seeks to the view-left instead of where the user tapped.

**Fix:** thread the body width from `LayoutBuilder` down to the tap
handler. Pass `pxPerMs` directly instead of reconstructing it.

```dart
// in _buildTrackBody
onTapDown: (d) => _handleBodyTap(d.localPosition, pxPerMs, tracks),
// then
void _handleBodyTap(Offset local, double pxPerMs, List<_TrackRow> tracks) {
  if (local.dy <= _rulerHeight) {
    final ms = (_viewLeftMs + local.dx / pxPerMs).round().clamp(0, _contentTotalMs);
    widget.onSeek(Duration(milliseconds: ms));
  }
}
```

Delete the `_pxToMs` method.

**Location:** nothing under `test/` for the new code.

**Problem:** Two pure-function modules are trivially testable and
shipped with zero coverage.

**Fix — two unit-test files:**

1. `test/subtitle_parser_test.dart` — covers:

   `FFmpegService._reduceToPeaks`:

Both files are pure Dart, no Flutter widget test harness needed.
Make `_reduceToPeaks` either public or add a `@visibleForTesting`
helper to expose it.

## Performance notes (not bugs, just future watchpoints)

- **`_contentTotalMs` is O(n) per build.** Runs twice per build inside
  `VideoDubTimeline.build`, which itself runs every ~33 ms (position
  tick). Fine for projects with hundreds of cues; start caching if
  a user imports 10k-cue auto-transcriptions.
- **`_MiniOverviewPainter` scans all clips + cues per repaint.**
  Same applies.
- **Waveform + ruler painters have proper `shouldRepaint` checks**
  and skip repaint when unchanged. ✅
- **Range scrubber drags re-layout the range-highlight rectangle
  on every pointer event** — that's correct, unavoidable.

## Security notes (not bugs)

- User-set ffmpeg path flows into `Process.run` with no sandboxing.
  User is trusted in their own install; no auto-download, so no
  supply-chain surface.
- FilePicker → `File.copy` destination goes through
  `PathService.dedupeFilename`, which sanitises the base name.
- All DB traffic is parameterised via drift.
