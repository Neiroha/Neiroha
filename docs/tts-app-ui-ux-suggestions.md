# 跨平台 TTS 软件功能调研与 UI/UX 优化建议

基于对目前市面上主流跨平台 TTS（Text-to-Speech）应用、AI 语音平台（如 Descript, ElevenLabs, Murf.ai, GPT-SoVITS 前端等）的调研，结合您提供的 UI 截图和设计构想，以下是对 Neiroha 项目的深入分析和功能建议：

---

### 1. 将 Quick TTS 与 Voice Bank 合并
**用户的想法**：将 Quick TTS 和 Voice Bank 合并，最右侧界面做成 Tab 切换（“编辑属性”与“快速测试”），方便添加音色后直接测试。
**调研与建议**：**非常赞同。**
* **减少上下文切换**：在当前的诸多主流工具中（如 ElevenLabs 的 Voice Library），用户在配置或查看一个特定 Voice 的同时，界面上通常会直接提供一个 Text 文本框用于“试听/生成”。
* **UI 布局**：左侧为音色列表，中间为选中音色的详细信息（大图/名称/描述），右侧面板可以做成顶部 Tab 分页（`Properties / Settings` 和 `Test / Quick TTS`）。这种设计能让工作流更闭环，用户在微调参数（如语速、情感）后可以立刻听效果。

### 2. 最下方的全局播放栏位置问题
**用户的想法**：最下面的播放栏是不是该放到上面去？
**调研与建议**：**视场景而定，建议解耦或吸底。**
* **全局 vs 局部**：如果播放的是单个片段（如 Quick TTS 里的一个音频），将播放控件直接做在对应条目上（Inline Playback）体验最好；如果是整个工程的预览，吸底播放栏（Bottom Bar）是业界的标准做法（如 Spotify, 甚至各类剪辑软件的屏幕底部）。
* **Timeline 冲突**：在 Phase TTS (有时间轴) 中，底部的全局播放栏可能会和 Timeline 争夺视觉重心和空间。建议在 Phase TTS 这种拥有复杂时间轴的界面中，将播放控制（播放/暂停/快进）**融合到时间轴的顶部工具栏上**，而在 Voice Bank 等简单界面保留底部全局播放状态。

### 3. Dialog TTS 的简化与聊天框 UI
**用户的想法**：Dialog TTS 没必要加 Timeline，做成右侧聊天框形式，导出时按顺序合成。
**调研与建议**：**非常棒且符合直觉的设计。**
* **聊天式排版**：播客或双人对话类文本，采用类似微信/Discord的左右气泡聊天框 UI，会让用户分配多角色音色时极其直观（气泡头像即为分配的 Voice）。
* **块状化生成 (Block-based)**：每一句话作为一个独立的 Block。不仅节省渲染算力（改一句只需要重新生成一句），管理起来也比 Timeline 简单得多。
* **导出能力**：提供两档导出——**合并导出**（自动按顺序拼接，可全局设置气泡间的静音间隔毫秒数）和**分段导出**（导出为一个 Zip，每句话命名为 `001_CharacterA.mp3`, `002_CharacterB.mp3`，极大方便游戏开发者和动画师）。

### 4. Phase TTS 的小说阅读与 AI 自动分段
**用户的想法**：针对长文本或小说，加入 AI 自动分段功能。
**调研与建议**：**这是长文本朗读的核心痛点，强烈建议加入。**
* **当前痛点**：手动将几万字的小说按段落复制进去或打断是非常痛苦的。
* **规则分段**：最基本需要支持按“回车/换行”、“句号/问号/感叹号”进行批量一键拆分。
* **AI/正则 角色提取 (Quote Extraction)**：这是一个极其杀手级的功能！通过正则（如自动提取双引号 `“ ”` 里的内容）或接通本地/云端 LLM，自动将文本拆分为【旁白】和【各角色对话】，并**自动为其分配不同的 Voice**。用户只需要做最后的确认和微调即可。

### 5. 新增 Video 视频配音模块（类简易版 PR）
**用户的想法**：参考 Phase TTS 加入视频播放，支持导入 LRC/SRT 自动生成配音并在时间轴上修改。
**调研与建议**：**极具商业价值的高级特性。**
* **工作流**：视频主区 + 底部时间轴 + 侧边字幕流。
* **导入字幕生成**：导入 SRT 后，软件将其转化为一排排的 Text Clips，直接调用 TTS 生成音频放在时间轴上。虽然一开始不用强制保证音画完全同步，但这已经能给漫剪、解说类 UP 主省去几十倍的时间。
* **未来拓展 (自动对齐)**：市面高级竞品（如 CapCut / 剪映）会根据字幕的时间戳，通过“音频时间拉伸 (Time-stretch)”算法让生成的 TTS 强行塞进视频原片段的时长内。前期你可以只做“按字幕生成和自由拖拽”（简易版 PR），之后再迭代高级对齐。

### 6. Voice Asset：构建私有语音克隆数据集
**用户的想法**：记录所有生成的和导入的音频，提供人工标记，用于导入语音克隆模型训练。
**调研与建议**：**极具差异化护城河的开发者/硬核功能。**
* **形成数据闭环**：一般的 TTS 只能“消耗”模型，如果你的软件能“反哺”模型，将是一个巨大的亮点。比如对接 GPT-SoVITS 时，用户在软件里积攒的高质量发音（或者用户自己上传的干音），可以在软件内通过 ASR（语音转写工具）自动打标。
* **校验 UI**：做一个网格或者列表视图（Audio list + Text input），用户听一句、核对一句字幕，没问题的勾选。最后提供一个 `Export Dataset` 按键，直接导出为 GPT-SoVITS 或 VITS 所需的 `.list` 格式目录结构。

### 7. 其他核心 TODO 功能建议
结合竞品调研，如果你要打造一个好用且功能完备的跨平台 TTS 工具，以下是可以逐步加入的 TODO：
1. **多音字匹配与读音修正 (Pronunciation & Dictionary)**：
   * 很多时候二次元名字或生僻字会读错。需要一个局部的 “多音字修正/拼音注音” 功能（类似SSML的 `<phoneme>` 标签），或者全局的**发音词典**（Find & Replace，比如看到“行长”固定读 `hang zhang`）。
2. **后期处理与特效 (Audio Effects)**：
   * 对生成的音频加一点基础的 DSP 处理极其加分。例如：给某句台词加点“混响（Reverb）”表示内心OS，加点“电话音/EQ”表示无线电通讯。
3. **批量替换音色 (Bulk Reassign)**：
   * 写小说经常中途觉得这个角色声音不合适，需要一键把该工程里所有“张三”的配音换成另一个 Voice Bank 模型，而不是一句句去改。
4. **统一的项目工程管理 (Project Saving)**：
   * 确保用户的段落、配音关系、时间轴可以保存为某种 `.neiroha` 工程文件或采用 SQLite 数据库存储，支持跨设备或备份。
5. **支持背景音乐 / 音效轨道 (BGM & SFX Track)**：
   * 如果你做了简单的多轨时间轴，可以允许用户拖入一首 MP3 作为背景音，并且在有 TTS 发音时支持**自动压频 (Auto-ducking)**。

---

**总结**：
您的规划方向非常准确，覆盖了目前 TTS 领域从轻量测试 (Quick TTS) -> 场景对话 (Dialog Chat) -> 长文本创作 (Phase TTS) -> 视频生产 (Video Dubbing) -> 数据反哺 (Voice Asset) 的一整套生态。建议优先落实 **合并 Quick TTS & Voice** 以及 **Dialog TTS 的聊天框 UI**，这两个改动成本较小但对整体体验提升巨大。
